    Potential Errors

    Common errors you may run into when setting up DynareJulia on Windows, and how to fix them. Click any error message below to expand the diagnosis and fix.

    UndefVarError: GeneralizedFirstOrderAlgorithm not defined **You see something like this when loading Dynare:** 
    ERROR: LoadError: UndefVarError: `GeneralizedFirstOrderAlgorithm` not defined
   @ C:\Users\<you>\.julia\packages\Dynare\<hash>\src\global\types.jl:150
in expression starting at .../Dynare.jl:1
ERROR: Failed to precompile Dynare ...
    **Why it happens.** Dynare 0.10.0 referenced a symbol (GeneralizedFirstOrderAlgorithm) that the NonlinearSolve package later renamed. If your Manifest.toml pins Dynare 0.10.0 but Pkg resolves the rest of the dependency tree against the newest NonlinearSolve, the symbol is gone and Dynare fails to precompile. This usually happens when a Manifest.toml was generated against an older Julia version (e.g. 1.9) and is now being used under Julia 1.10 or later. **Fix.** Wipe the lockfile and let Pkg re-resolve from the Project.toml. This pulls Dynare 0.10.4 (or whatever the latest compatible version is) with a matching set of dependencies. In a PowerShell terminal at the project root: 
    # 1. Activate the project (sets JULIA_PROJECT to .\venv)
. .\activate.ps1

# 2. Delete the lockfile
Remove-Item .\venv\Manifest.toml

# 3. Re-resolve and precompile
julia -e "using Pkg; Pkg.instantiate(); Pkg.precompile(); Pkg.status()"
    After this finishes, Pkg.status() should report: 
    [a93c6f00] DataFrames v1.8.2
[5203de40] Dynare v0.10.4    <-- was 0.10.0 before the fix
[91a5bcdd] Plots v1.41.6
[bd369af6] Tables v1.12.1
    If Dynare is still listed as 0.10.0, your Project.toml has a hard pin — remove the [compat] Dynare line and try again.
    cannot remove ... Access is denied [stockflow/model/json] **You see this when running a model:** 
    Starting preprocessing of the model file ...
terminate called after throwing an instance of 'std::filesystem::__cxx11::filesystem_error'
  what():  filesystem error: cannot remove: Access is denied [stockflow/model/json]
ERROR: LoadError: type Array has no field results
    **Why it happens.** Two things together:
    • The Dynare preprocessor wants to delete the output folder from a previous run before writing fresh output.
    • Dropbox or OneDrive holds file locks on those files while syncing them. The "Access is denied" is the sync client refusing to let go.
    When the preprocessor crashes, @dynare returns an empty Vector{Any} instead of a Context — that's why the next line errors with "type Array has no field results". **Fix.** Two options, in order of preference: Move the project out of Dropbox. Put your Codes\ folder somewhere like C:\dev\thailand2026\. Keep slides and PDFs in Dropbox; keep the code out. DSGE solver output churns hundreds of small files per run, and Dropbox sync hates that. Or pause Dropbox before running. Right-click the Dropbox tray icon → Pause syncing → run your model → resume. If you control the driver script, you can also add a safe_rm() helper that retries with backoff — it papers over transient locks but not persistent ones.
    Failed to precompile GR_jll / Plots / Dynare **You see a cascade of precompile failures:** 
    Precompiling project...
  ✗ GR_jll
  ✗ Qt6ShaderTools_jll
  ✗ GR
  ✗ Plots
  ✗ StatsPlots
  ✗ Dynare
  N dependencies errored.
    Notice the cascade: GR_jll is the root failure; everything else fails because it depends on Plots which depends on GR. **Why it happens.** Almost always Windows long-path limits. The default MAX_PATH is 260 characters. A depot path like: 
    C:\Users\asyak\Dropbox\Work\The Better Policy Project\Courses\Thailand 2026\DSGE - Week 2\Codes\.julia\artifacts\<40-char-hash>\bin\gr\fonts\...
    is already 165 characters before GR's nested folder structure starts. GR routinely overflows it. **Fix.** Use a short depot path that lives outside Dropbox.
    1. Edit activate.ps1 to set:
    $env:JULIA_DEPOT_PATH = "C:\jl-depots\thailand2026"
    1. Re-activate and re-instantiate:
    . .\activate.ps1
julia -e "using Pkg; Pkg.instantiate(); Pkg.precompile()"
    The Project.toml and Manifest.toml stay in venv\ (small text files, fine in Dropbox). Only the multi-GB depot moves to a short path on the system drive. Alternative: enable Windows long-path support globally. Open gpedit.msc → Computer Configuration → Administrative Templates → System → Filesystem → "Enable Win32 long paths" → Enabled. Requires admin and a reboot.
    ArgumentError: invalid index ! when extracting IRFs **You see one of these when plotting impulse responses:** 
    ERROR: ArgumentError: invalid index: ! of type typeof(!)
    or: 
    MethodError: no method matching expand_extrema!(::Plots.Axis,
  ::AxisArrayTables.AxisArrayTable{Float64, ...})
    **Why it happens.** DynareJulia returns IRFs as AxisArrayTable, not DataFrame. DataFrame indexing syntax (tbl[!, :col] or tbl[:, :col]) does not work — ! is a function, and : returns a one-column 2D table that Plots can't accept as a y-series. **Fix.** Use the Tables.jl column accessor and convert to a plain vector: 
    using Tables
series = collect(Float64, Tables.getcolumn(irfs[shock], :b))
    This works whether the underlying type is DataFrame, AxisArrayTable, or anything else implementing the Tables.jl interface — so the script stays robust across Dynare versions.
    activate.ps1 : The term '.\activate.ps1' is not recognized **You see this in PowerShell:** 
    activate.ps1 : The term 'activate.ps1' is not recognized as the name of a
cmdlet, function, script file, or operable program.
    **Why it happens.** PowerShell will not run a script in the current folder unless you prefix it with .\. And to keep the environment variables in your shell (rather than throwing them away in a child process), you also need to dot-source the script with a leading .  (period + space). **Fix.** Run it with both prefixes: 
    . .\activate.ps1
    That is: period — space — period — backslash — activate.ps1. Four characters before the filename. If you also see "running scripts is disabled on this system", run this once per machine: 
    Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

    Working setup reference

    The two-command workflow once everything is in place:

    cd "C:\...\Codes"
. .\activate.ps1
julia run_stockflow.jl

    activate.ps1 sets JULIA_PROJECT to .\venv (and, if you applied the long-path fix above, JULIA_DEPOT_PATH to a short path).

    A successful run ends with output like:

    ================ Steady state ================
  b        =   0.500000
  def      =   0.014563
  prdef    =  -0.004854
  intpay   =   0.019417
  y        =   0.000000
